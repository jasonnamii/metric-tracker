# 프록시 지표(Proxy Metrics) — 언제 작동하고, 언제 실패하는가

중요하지만 **직접 측정 불가능한 것**을 간접 지표(프록시)로 대체할 때 반드시 알아야 할 것.

---

## 1. 프록시 지표의 정의와 필요성

### 정의

**진정한 구성개념(construct)을 직접 측정할 수 없을 때, 그것을 대리하는 관찰 가능한 지표.**

### 필요한 이유

일부 중요한 것들은 정량화 불가능 (또는 매우 비쌈):

| 진정한 구성개념 | 직접 측정이 어려운 이유 | 프록시 지표 |
|-------------|------------------|---------|
| **고객 만족도** | 정의가 모호하고, 주관적이며, 시간에 따라 변함 | NPS, CSAT, 리뷰 점수 |
| **제품 품질** | "좋다"는 것을 어떻게 정의할 것인가 | 버그율, 성능 지표, 테스트 커버리지 |
| **직원 헌신도(Engagement)** | 마음 상태는 측정 불가능 | 퇴사율 역수, 급여 인상 요청 비율, 사내 이벤트 참석율 |
| **혁신 능력** | 아직 나타나지 않은 미래 결과 | R&D 투자, 특허 출원, 새 기능 출시 빈도 |
| **신뢰도(Trust)** | 심리 상태 측정 불가 | 규정 준수율, 고객 에러 후 재가입율, NPS 상승 |

---

## 2. 프록시가 작동하는 조건 (5가지)

프록시가 유효하려면:

### 1) 인과 이해 (Causal Understanding)

프록시와 진정한 구성개념 사이의 **인과 메커니즘을 명확히 알고 있어야 함.**

**예: NPS (Net Promoter Score)**

```
설문: "이 회사를 친구에게 추천하겠는가? (0~10)"

인과:
- 고객이 10점 주는 이유? = 제품이 기대치를 넘어섰기 때문 (만족도)
- 고객이 0~6점 주는 이유? = 제품이 기대치 못 미쳤기 때문 (불만)

따라서: NPS ↑ = 고객 만족도 ↑ (인과 있음)
```

**반례: 직원 결근율을 "직원 행복도"의 프록시로**

```
인과:
- 결근 ↓ = 행복도 ↑? (불완전함)
- 결근의 다른 이유: 돈이 부족해서 병가 남기지 않으려 함, 
                     또는 상사가 결근 통보를 어려워함

따라서: 약한 프록시. 보조 지표로만 사용 권장
```

### 2) 안정적 상관성 (Stable Correlation)

프록시와 구성개념의 관계가 **시간과 상황에 따라 변하지 않아야 함.**

**예: YouTube의 "Watch Time"**

```
프록시: Watch Time (사용자가 본 총 시간)
구성개념: 사용자 만족도

초기 상관성: Watch Time ↑ = 만족도 ↑ (강한 상관, r = 0.85)

10년 후: 추천 알고리즘 변화
- Watch Time은 계속 ↑ (알고리즘이 중독성 콘텐츠 추천)
- 만족도는 → (고객들은 "시간 낭비" 느낌)

결론: 상관성 약화 (r = 0.4). 프록시로서 신뢰도 하락
```

### 3) 조작 어려움 (Gaming Resistance)

프록시를 직접 올리려고 해도, **쉽게 조작되지 않아야 함.**

**좋은 프록시**: NPS
```
NPS를 올리려면?
  - 고객을 속일 수 없음 (설문 조작 탐지됨)
  - 진정한 고객 만족 개선만 가능
  
결론: 게임화 어려움
```

**나쁜 프록시**: 고객 지원 응답 시간
```
응답 시간을 올리려면?
  - 고객 설명 무시하고 성급한 회신
  - 또는 "자동 응답"으로 기술적으로 시간 단축
  
결론: 게임화 쉬움 (질 악화). 프록시 신뢰도 빠르게 하락
```

### 4) 정기 검증 (Regular Validation)

프록시의 신뢰도는 **시간에 따라 자연스럽게 하락함.** 정기적으로 재검증 필수.

**검증 방법**:

| 주기 | 방법 | 확인 사항 |
|------|------|---------|
| **분기** | 고객 인터뷰 10명 | NPS와 실제 만족도 일치하는가? |
| **반년** | 샘플 분석 | 프록시 상위층(NPS 9~10)이 실제로 재구매율도 높은가? |
| **1년** | 통계 상관도 재계산 | r값이 여전히 0.7 이상인가? |

### 5) 실패 신호 정의 (Failure Signal)

프록시를 언제 폐기할 것인지 **미리 정의해야 함.**

```
"우리는 NPS를 고객 만족도 프록시로 사용한다.
 
다음 중 하나가 발생하면 즉시 폐기하고 재평가한다:
 
1. 상관도 r < 0.6 (분기 재검증에서)
2. NPS 점수는 높은데, 고객 인터뷰에서 "시간 낭비"라는 피드백 (정성 괴리)
3. 고객 이탈율이 증가하는데 NPS는 유지/상승 (역행)
4. 경쟁사의 NPS가 우리보다 10점 낮은데 시장점유율은 높음 (맥락 변화)
"
```

## 3. 프록시 실패의 5가지 패턴

### 패턴 1: 단절/기관상실 (Disconnect & Institutional Amnesia)

**원인**: 프록시를 설계한 사람이 떠나고, 새 사람은 원래 이유를 모름.

**진행**:

```
Year 1: 프록시 설계자가 명확히 정의
        "우리는 X를 Y로 측정한다. 이유는..."

Year 2: 설계자 떠남. 문서 남음.

Year 3: 신입 직원들은 문서 안 읽음.
        "지표가 있으니까 하는 대로 따라하지"

Year 4: 원래 의도와 관계없이 지표 최적화만 반복
```

**신호**: 
- "왜 우리가 이 지표를 추적하는가?"라는 질문에 답 못 함
- 프록시 정의가 여러 버전으로 변형

**방어**:
- 프록시 설계 시 **설계 근거를 문서화**
- 분기마다 "왜 이 프록시인가?" 재확인 회의
- 신입 온보딩 시 프록시 배경 교육 필수

### 패턴 2: 게임화/신호 침식 (Gaming & Signal Erosion)

**원인**: 초반엔 프록시가 작동하지만, 점차 게임화됨.

**진행**:

```
Phase 1 (초기, 3개월):
  지표 올리는 노력 = 진정한 개선
  예: 고객 응답 시간 단축 = 팀이 효율성 개선

Phase 2 (중기, 6개월):
  지표 올리는 방법이 다양화
  예: 우선순위 재조정으로 인해 일부 고객 대기시간 증가
       (측정 고객층이 바뀜)

Phase 3 (후기, 12개월):
  지표 조작 시작
  예: 자동 응답으로 기술적으로 응답시간 단축
       (진정한 고객 만족은 악화)

Phase 4 (극단화):
  지표와 구성개념 완전 역행
  예: 응답 시간은 5분, 하지만 고객 불만족 최고조
       (답변 내용은 무관)
```

**신호**:
- 지표는 올랐는데, 정성적 피드백("문제는 여전하다") 나옴
- 지표의 계산 정의가 자주 변경
- 팀이 창의적인 방법으로 지표 올리기 시작

**방어**: → §5. 프록시 검증 프로토콜 참조

### 패턴 3: 측정-신뢰도 균형 (Legibility-Fidelity Tradeoff)

**원인**: 간단한 프록시는 측정하기 쉽지만, 정확도가 낮음.

**스펙트럼**:

```
간단함 ←───────────────────→ 정확함

NPS ◄────── 측정 용이, 명확 ──────►
  (1개 질문, 분석 쉬움)
  (하지만 정확도는 60~70%)

Customer Effort Score (CES) ◄─────────────────────►
  (간단하지만, 무엇을 의미하는가 불명확)

고객 생명주기 가치(LTV) ◄──────────────────────────────────────►
  (매우 정확, 하지만 계산 복잡, 구성이 여러 요소 혼합)
```

**딜레마**:

| 선택 | 장점 | 단점 |
|------|------|------|
| **간단한 프록시 선택** | 모두가 이해하고 추적 용이 | 조작되기 쉬움, 정확도 낮음 |
| **복잡한 프록시 선택** | 정확하고 조작 어려움 | 계산 복잡, 설명 어려움 |

**해결책**: 
- 처음엔 간단한 프록시로 시작 (NPS)
- 6개월 후 검증 (정확도 낮으면 복잡한 프록시 추가)
- 둘을 혼합 사용 (NPS + 이탈율 같이 추적)

### 패턴 4: 범위 제한 (Range Limitations)

**원인**: 프록시는 특정 범위에서만 유효하고, 그 밖에서는 깨짐.

**예: Productivity 지표로 "코드 라인 수(LOC)"**

```
초기 조건 (작은 팀, 30명):
  코드 라인 ↑ = 개발 생산성 ↑ (대략 맞음)

성장 후 (큰 팀, 300명):
  코드 라인이 의미하는 것?
  ├─ 새 기능 (생산성 ↑)
  ├─ 기술 부채 (생산성 ↓)
  ├─ 코드 중복 (생산성 ↓)
  └─ 불필요한 주석 (상관없음)

결론: 크기가 커지면서 프록시 신뢰도 하락
```

**신호**:
- 조직 규모가 바뀐 후 지표의 해석이 모호해짐
- 팀 구성이 바뀐 후 프록시 추세와 실제 성과가 괴리

**방어**:
- 프록시 설계 시 **유효 범위를 명시** ("30~100명 팀에서만 유효")
- 범위 벗어나면 재검증 또는 폐기
- 계절별/지역별/세그먼트별로 따로 검증

### 패턴 5: 인과 혼동 (Causal Confusion)

**원인**: 상관관계를 인과관계로 착각.

**예: 마케팅 이메일 클릭율**

```
상관도 r = 0.85 (매우 높음):
  이메일 클릭율 ↑ ↔ 구매율 ↑

하지만 인과는?

가설 A (프록시로 작동):
  이메일 클릭 ↑ → 제품 관심 ↑ → 구매 ↑

가설 B (역인과):
  이미 관심 고객이 (자주) 이메일 클릭
  → 상관은 높지만, 클릭이 구매를 일으키지 않음

가설 C (공통원인):
  제품이 좋아져서 (원인)
  → 고객이 (더 자주) 이메일 클릭
  → 고객이 (더 많이) 구매
  → 클릭과 구매는 모두 "제품 개선"이 원인

결론: A, B, C 중 어느 것이 맞는가?
       프록시로 사용하려면 인과 확실히 해야 함
```

**신호**: 
- 지표는 높은데 결과는 악화
- "왜 이 지표가 다른 지표와 상관있는가?"라는 질문 난처함

**방어**: 
- 상관도 r > 0.7은 필수이지만, 충분하지 않음
- 인과 메커니즘을 **명확히 가설화하고 검증**
- 역인과 여부 점검

## 4. 프록시 검증 프로토콜 (3단계)

모든 프록시 지표 설계 시:

### 단계 1: 명시적 선언

```
"우리는 _____(진정한 구성개념)을(를) 측정하려고 한다.

프록시: _____(관찰 가능한 지표)

선택 근거: _____(인과 메커니즘)

한계: _____(이 프록시의 한계, 측정 안 하는 것)

검증 방법: _____(프록시 신뢰도를 어떻게 확인할 것인가)

폐기 신호: _____(언제 이 프록시를 버릴 것인가)
"
```

**예시**:

```
프록시: NPS (Net Promoter Score)
구성개념: 고객 만족도
인과: 9~10점 고객 = 제품이 기대치 넘춤 = 재구매 가능성 높음
한계: NPS는 "추천 의도"만 측정, 실제 사용도와는 별개
검증: 분기마다 고객 인터뷰 10명, 실제 만족도와 NPS 점수 비교
폐기 신호: (1) r < 0.6 (2) NPS 상승인데 이탈율 증가 (3) 경쟁사 NPS 훨씬 낮은데 시장점유율 높음
```

### 단계 2: 정기적 재검증

| 주기 | 검증 항목 | 책임자 |
|------|---------|--------|
| **월간** | 지표 추세 확인. 예상과 다른가? | 데이터 담당 |
| **분기** | 샘플 검증. NPS 9~10점 고객의 재구매율은? | 카테고리 리더 |
| **반년** | 상관도 재계산 (r값) | 데이터 분석팀 |
| **1년** | 고객 집단 인터뷰 (정성 검증) | CMO 또는 제품 리더 |

### 단계 3: 실패 신호 대응

프록시 신뢰도가 떨어지면:

```
신호 1개만 나타나면: "경고" 상태
  └─ 조사 시작, 하지만 지표는 계속 추적

신호 2개 이상:
  └─ 즉시 재평가 회의 소집
  └─ 폐기 또는 재정의

신호 재확인:
  └─ 위양성(false positive)인지 확인
  └─ 맥락 변화인지 재정의 필요인지 판단
```

## 5. 프록시 설계 체크리스트

새 프록시 지표를 만들 때:

| 확인 | 예/아니오 | 비고 |
|------|---------|------|
| 진정한 구성개념이 무엇인가? | ◻ | 명확히 정의 필수 |
| 인과 메커니즘을 설명할 수 있는가? | ◻ | "왜 이것이 구성개념을 대리하는가?" |
| 이미 알려진 상관도 r > 0.7인가? | ◻ | 문헌 또는 초기 데이터 확인 |
| 지표를 쉽게 조작할 수 있는가? | ◻ | "예"면 더 나은 프록시 찾기 |
| 검증 방법을 정했는가? | ◻ | 월/분기/반년/1년 주기 명시 |
| 폐기 조건을 정했는가? | ◻ | "3개 신호 = 폐기" 같은 명확한 규칙 |
| 한계를 문서화했는가? | ◻ | "이것이 측정하지 않는 것은..." |

---

## 요약: 프록시 지표 대원칙

1. **필요성**: 직접 측정 불가능한 중요한 것을 측정하려 할 때만 사용
2. **명시성**: 프록시와 구성개념의 관계를 명확히 선언
3. **검증성**: 정기적으로 신뢰도를 재확인 (r값, 정성 피드백)
4. **폐기 준비**: 미리 "언제 버릴 것인가"를 정의
5. **인과 확인**: 상관은 높지만 인과는 확실한가? 재확인

**핵심**: 프록시는 도구일 뿐, 목표가 아니다. 프록시가 약해지면 즉시 갈아타기.
