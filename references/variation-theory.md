# 변동유형 진단 — Shewhart, Deming, 공통원인 vs 특수원인

모든 프로세스에는 변동이 있다. **변동의 종류를 제대로 진단하지 못하면 반대 방향으로 대응한다.** (Deming, 1986)

---

## 1. Shewhart 관리도(Control Chart) 기초

### 역사: Bell Labs, 1924

Walter Shewhart는 전화 부품 제조 품질 관리를 위해 관리도를 발명. 이후 모든 통계적 공정 관리의 기초가 됨.

### 구조

```
개별 측정값 플롯
     ┃
  UCL├─ ─ ─ ─ ─ ─ ─ ─
     ┃ ┌─    ┌─
  중심├─ ─ ─ ─ ─ ─ ─ ─ (평균)
     ┃ └─    └─
  LCL├─ ─ ─ ─ ─ ─ ─ ─
     ┃
   시간
```

- **UCL (Upper Control Limit)** = 평균 + 3σ (표준편차)
- **중심선** = 과거 평균
- **LCL (Lower Control Limit)** = 평균 - 3σ

### 판정 규칙

| 신호 | 의미 | 대응 |
|------|------|------|
| **점이 한계 밖** (UCL 초과 또는 LCL 이하) | 특수원인 변동 | 근본원인 찾기 |
| **8~9개 점이 중심선 한쪽** | 추세 시작 | 시스템 변경 감지 |
| **점들이 중심 부근 내에서 무작위** | 공통원인 변동 | 시스템 재설계 필요 |

## 2. 공통원인(Common Cause) 변동

### 정의

**시스템에 내재된, 만성적, 항상 존재하는 변동.**

모든 프로세스는 공통원인 변동을 가지고 있다. 예:

- 제조업: 기계 정밀도 한계, 재료 편차, 환경 온도 변화
- 영업: 대고객 운수, 계절성, 시장 사이클
- 소프트웨어: 개발자 숙련도 편차, 요구사항 모호성, 테스트 범위 한계

### 신호

```
관리도에서 모든 점이 UCL-LCL 사이에 있고, 무작위로 분포
```

### 관리자의 오류 대응 (Deming의 깔때기 실험)

| 잘못된 대응 | 결과 |
|-----------|------|
| "이번 달은 떨어졌네, 팀을 혼낸다" | 다음 달은 우연히 올라감 → 혼냈던 게 맞는 줄 착각 |
| "지난달에 성공했으니, 그 방법을 계속 써" | 우연히 성공했던 것 재현 못함 → "왜 안 되지?" 혼란 |
| "새 인센티브를 도입하면 변동이 줄어들겠지" | 변동은 오히려 증가 (시스템이 더 불안정해짐) |
| "분기마다 다른 전략으로 바꿔보자" | 변동 증가, 혼란 가중 |

### 유일한 해결책: 시스템 재설계

공통원인을 줄이려면 **시스템 자체를 바꿔야 한다.** 단기 조치로는 불가능:

| 영역 | 공통원인 감소 방법 |
|------|------------------|
| 제조 | 기계 업그레이드, 프로세스 표준화, 자동화 |
| 영업 | 고객 세분화, 타이밍 최적화, 제품 개선 |
| 소프트웨어 | 아키텍처 개선, 테스트 자동화, 기술 부채 제거 |

### 신호: 공통원인으로 착각하면 안 되는 경우

| 신호 | 실제는 | 대응 |
|------|------|------|
| 모든 점이 범위 내 + 한 점만 급격히 올라감 | 특수원인 | 근본원인 분석 |
| 점들이 중심 한쪽에 계속 쏠림 (8개 연속) | 추세 = 시스템 변경 신호 | 무엇이 변했는가 찾기 |

## 3. 특수원인(Special Cause) 변동

### 정의

**식별 가능한, 비정상적, 임시적인 원인에서 비롯된 변동.**

예:

- 제조: 기계 고장, 새로운 작업자, 재료 로트 불량
- 영업: 대형 고객 한 명 나감, 경쟁사 가격 인하, 제품 결함
- 소프트웨어: 보안 패치, 인프라 다운, 신입 개발자 합류

### 신호

```
한 점이 UCL/LCL 밖에 있거나,
몇 개 점이 갑자기 다른 위치에서 시작
```

### 올바른 대응

**특수원인은 식별 가능하고 제거/교정 가능:**

| 영역 | 특수원인 제거 |
|------|-------------|
| 제조 | 고장난 기계 수리, 불량 재료 폐기, 작업자 재교육 |
| 영업 | 고객 복원, 경쟁사 대응, 제품 개선 |
| 소프트웨어: 긴급 패치, 인프라 복구, 온보딩 강화 |

## 4. Deming의 심오한 지식의 체계(System of Profound Knowledge)

공통원인 vs 특수원인을 제대로 구분하려면 4개 영역이 서로 연결되어야 한다:

### ①시스템 이해 (Appreciation of System)

- 조직은 **상호 연결된 부분들의 체계**
- 개별 부분 최적화 ≠ 전체 최적화
- 예: 영업 팀의 노력 ↑ → 부실납품 급증 → 전체 이익 ↓

**지표 설계 함의**: L2 드라이버를 최적화할 때, 다른 L2 지표와의 상호작용을 확인해야 함.

### ②변동 이론 (Knowledge of Variation)

- 공통원인과 특수원인을 구분하지 못하면 모든 대응이 역효과
- "좋은 결과"는 운일 수 있고, "나쁜 결과"는 최적 노력일 수 있음
- Shewhart 관리도: 판단의 유일한 객관 기준

**지표 설계 함의**: 리뷰 시 변동유형 진단 필수 (SKILL.md §4 ②)

### ③지식 이론 (Theory of Knowledge)

- 관리 = 예측 기반 행동
- 예측은 이론(가설)을 기반으로 함
- 관찰·실험·검증 사이클 필수

**지표 설계 함의**: "이 지표 올리면 L1도 올라간다"는 가설을 명시 + 주기적 검증

### ④심리학 (Psychology)

- 내재 동기(intrinsic motivation) > 외재 동기(extrinsic)
- 공포 기반 관리(fear-based) ⇒ 변동 증대
- 신뢰, 자율성, 의미 추구가 성과의 진정 동인

**지표 설계 함의**: 지표로 보상을 결정하면 (외재 동기화) 심리학적 악순환 발생

## 5. KPI 리뷰 의사결정 트리

### 메트릭이 변했을 때의 판정 절차

```
지표값이 전월 대비 변했다
    ↓
기준값(기준선) ± 3σ 범위 내?
    │
    ├─ YES → 공통원인 변동
    │   └─ 행동: 시스템 재설계만 가능. 단기 조치 금지
    │
    └─ NO → 특수원인 변동
        └─ 행동: 근본원인 분석 → 표적 개입
```

### 분기 리뷰 판정

```
분기 평균 vs 전분기 평균 비교
    ↓
통계적으로 유의미한 차이?
(일반적으로 ±15% 이상)
    │
    ├─ NO → 공통원인
    │   └─ 전략 유지
    │
    └─ YES → 특수원인 추정
        ├─ 명확한 원인 찾았는가?
        │   ├─ YES → 원인 제거 또는 유지 판단
        │   └─ NO → "확인필요" 태그, 다음 주 심화 조사
        │
        └─ 원인 모름
            └─ 추가 데이터 수집 (1주 대기)
```

### 실행 팀 입장에서의 판정

| 상황 | 판정 | 대응 |
|------|------|------|
| "지표 떨어졌다고 했는데, 범위 내?" | 공통원인 | "좋았던 때는 운이었고, 이것도 운. 시스템 개선 계획 없으면 기대 말 것" |
| "지표 올랐고 범위 밖. 뭐 했냐?" | 특수원인 | "A를 했다" → 근본원인 확인 → 재현성 확인 |
| "지표 계속 내려가는데..." | 추세 감지 | "8주 연속 하향? 시스템 근본 변경 감지. 실행 팀의 문제 아님, 외부 환경 또는 전략 변경 필요" |

## 6. 관리자 판단 오류의 3가지 유형

### 제1종 오류 (Type I): 공통원인을 특수원인으로 착각

**상황**: 정상 범위 내 변동인데 "특수한 원인이 있을 것"이라 가정 → 범인 찾기 시작

**결과**: 억울한 담당자 질책 + 불필요한 개선 시도 → 시스템 불안정화

**예시**:
- "이번 달 판매가 10% 떨어졌으니 영업 팀장을 바꿔보자"
- "버그가 3개 있으니 테스트 팀에 문제가 있다"
- 실제: 이 정도 변동은 매달 있는 일. 원인 없음.

**비용**: 우수 인재 손실, 팀 모랄 저하, 시스템 불안정화

### 제2종 오류 (Type II): 특수원인을 공통원인으로 착각

**상황**: 명백한 특수원인(기계 고장, 큰 고객 손실)인데 "별것 아니다" 무시

**결과**: 문제 악화, 근본원인 미해결 → 누적 피해

**예시**:
- "이번 달 분기 목표 미달했지만, 내년에 회복되겠지"
- 실제: 핵심 고객이 나감. 내년에 회복 불가능.

**비용**: 시장점유율 감소, 신뢰 상실

### 제3종 오류 (Type III): 추세를 공통원인으로 착각

**상황**: 8개월 연속 하향 추세인데 "변동 범위 내니까 괜찮다" 무시

**결과**: 시스템 근본 문제 미해결 → 전략적 위기

**예시**:
- "월간 지표 변동 ±20%이니 이 추세는 정상"
- 실제: 시장 침식, 경쟁사 추격, 제품 노후화 신호

**비용**: 사업 위기

## 7. 실전: 업종별 변동유형 판정 사례

### 영업 매출 리뷰

```
월간 매출 변동 ±15% 범위 내:
  └─ 공통원인 → 영업 전략 자체는 유지하되,
               시스템 개선 (고객 세분화, 타이밍 최적화 등)
               
월간 매출 1년 평균 대비 20% 이상 하락:
  └─ 특수원인 → 근본원인 분석
      "큰 고객 나갔나?" → YES → 복구 시도
      "경쟁사 가격 인하?" → YES → 대응 전략
      "제품 결함?" → YES → 즉시 개선
      "이유 모름?" → 추가 조사 (영업팀 인터뷰)
```

### 소프트웨어 버그 리뷰

```
주간 버그 5~10개 (정상 범위):
  └─ 공통원인 → 아키텍처 개선, 테스트 자동화, 코드리뷰 강화
  
주간 버그 25개 (범위 3배):
  └─ 특수원인 → 근본원인 분석
      "새 기능 배포?" → 기능 품질 재검토
      "신입 온보딩?" → 페어링, 멘토링 강화
      "알려진 버그 폭증?" → 기술 부채 처리
```

### 고객 이탈률 리뷰

```
월간 이탈률 3~5% (정상):
  └─ 공통원인 → 고객 리텐션 프로그램 개선
               (가치 제시, 만족도 향상, 온보딩)
  
월간 이탈률 15% (3배 이상):
  └─ 특수원인 → 즉시 근본원인 분석
      "제품 장애?" → 복구 + 고객 보상
      "경쟁사 침탈?" → 고객 재계약
      "가격 인상?" → 커뮤니케이션 개선 또는 정책 검토
      "핵심 팀 퇴사?" → 지식 전수, 고객 업데이트
```

## 8. Gotchas: Deming의 깔때기 실험

### 시나리오

구슬을 깔때기에 떨어뜨려 목표점에 착지시키는 게임.

```
목표점
   ▲
   │●(구슬)
   │
━━━━━━ (깔때기)
```

### 규칙 변형

| 규칙 | 결과 |
|------|------|
| **규칙 1: 제한 없음** | 구슬 다양하게 떨어짐 (공통원인 변동) |
| **규칙 2: "다음엔 이번보다 정정해야 한다" 생각하고 깔때기 조정** | 변동 **증가** |
| **규칙 3: "이번에 성공했으니 같은 조정" 계속** | 변동 **급증** |
| **규칙 4: "아예 다른 방법" 시도하기 | 변동 **최악** |

### 교훈

**공통원인에 대한 조정은 변동을 증대시킨다.** (tampering)

이것이 조직에서 벌어지는 일:
- "지난달 실패했으니 전략 변경"
- "저 사람 분기 실패했으니 팀 이동"
- "지표 떨어졌으니 인센티브 재설계"

결과: 변동 더 심함.

## 요약: 리뷰 전 체크리스트

매주 리뷰 시작 전에 확인:

| 지표 | 기준선 | 현재값 | 범위 내? | 판정 | 대응 |
|------|--------|--------|---------|------|------|
| 매출 | $100k | $92k | Yes (범위 $80-120k) | 공통원인 | 전략 유지 |
| 이탈률 | 4% | 12% | No | 특수원인 | 근본원인 분석 |
| 버그 | 7/주 | 8/주 | Yes | 공통원인 | QA 프로세스 개선 |

**결론**: 판정이 먼저. 판정 없이 행동하면 역효과.
