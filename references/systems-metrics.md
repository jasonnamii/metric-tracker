# 시스템과 지표 — Meadows, 시스템 다이내믹스, 제약이론

지표 설계는 시스템을 이해하는 것부터 시작해야 한다. 잘못된 지표는 시스템을 이해하지 못한 증거다.

---

## 1. Donella Meadows의 12단계 레버리지 포인트

Meadows는 시스템의 어느 부분에 개입하는지에 따라 변화 효과가 달라진다고 주장:

### 레버리지 낮음 (1~5단계): 증상 치료, 효과 미미

| 순서 | 레버리지 | 설명 | 예시 | 지표 설계의 의미 |
|------|---------|------|------|-----------------|
| **1** | 상수(Constants) | 시스템 파라미터 조정 | "재고 최소량을 10에서 12로" | 가장 낮은 효과. 대부분의 조직 메트릭이 여기 해당 |
| **2** | 버퍼(Buffers) | 시스템의 용량 증가 | "창고 크기를 2배로" | 일시적 효과만 있음 |
| **3** | 정보 흐름(Information flows) | 누가 어떤 정보를 아는지 | "마케팅이 실제 고객 피드백을 본다" | 개선 가능하지만 천천함 |
| **4** | 규칙(Rules) | 행동의 제약조건 | "판매자에게 원가 정보 공개" | 부분적 개선 |
| **5** | 규칙 변경 권력(Power to change rules) | 규칙을 바꿀 권한 | "팀이 자기 목표를 설정한다" | 조직 문화 변경 필요 |

**문제**: 대부분 조직이 1~5단계에서만 지표를 추적. 상수, 버퍼, 보상 구조 조정만 시도.

### 레버리지 중간 (6~8단계): 구조 변경, 높은 효과

| 순서 | 레버리지 | 설명 | 예시 | 지표 설계의 의미 |
|------|---------|------|------|-----------------|
| **6** | 피드백 루프(Feedback loops) | 정보가 행동에 영향 주는 방식 | 가입 직후 가치 경험 → 전환율 ↑ | 루프의 강도를 측정하는 지표 필요 |
| **7** | 시스템 목표(System goal) | 시스템 전체가 최적화하는 것 | "단기 이익 극대화" vs "고객 만족도" | 목표 불일치 = 지표 불일치 |
| **8** | 패러다임(Paradigm) | 시스템의 기본 세계관 | "주주가치 극대화" vs "이해관계자 가치" | 지표 선택 자체가 패러다임을 반영 |

### 레버리지 높음 (9~12단계): 근본 변경, 극한 효과

| 순서 | 레버리지 | 설명 | 예시 | 지표 설계의 의미 |
|------|---------|------|------|-----------------|
| **9** | 자기 조직화 능력(Self-organization) | 시스템이 스스로 진화할 수 있는가 | 새로운 역할, 조직 구조 출현 | 변화 속도 자체를 측정 |
| **10** | 정보 구조 변경(Information structure) | 기본 인과관계 재정의 | "비용 중심 회계" → "활동기반 원가계산" | 회계 시스템 자체 변경 |
| **11** | 규칙 간 계층 변경(Hierarchy) | 의사결정 권한 구조 | 중앙집중식 → 분권화 | 의사결정 속도 자체가 지표 |
| **12** | 패러다임 초월 (Transcend) | 근본적 세계관 변경 | "성장" 아이디어 자체 포기 → "안정성" 추구 | 가장 어렵고, 지표로 측정 불가능에 가까움 |

### 지표 설계 함의

```
당신이 추적하는 지표의 레버리지 포인트가 무엇인가?

L2 드라이버 지표 대부분:
  → 레버리지 1~3단계 (상수, 버퍼, 정보 흐름)
  → 단기 효과 있으나, 시스템 문제 근본 해결 불가

시스템 다이내믹스 감시 필수:
  → 6~8단계 피드백 루프와 목표 정렬 확인
  → L3·L4 카운터 지표로 의도하지 않은 결과 감시
```

## 2. 시스템 다이내믹스 기초

### 핵심 개념

시스템은 다음 4가지로 이루어짐:

| 개념 | 정의 | 예시 |
|------|------|------|
| **Stock (스톡)** | 시스템 내에 축적되는 것 | 고객, 현금, 인벤토리, 직원 |
| **Flow (플로우)** | 스톡을 증가/감소시키는 속도 | 신규 가입수/월, 이탈수/월, 매출/월 |
| **Feedback Loop** | 피드백이 미래 플로우에 영향을 주는 방식 | 고객 ↑ → 수익 ↑ → 제품 투자 ↑ → 고객 ↑ |
| **Delay (지연)** | 원인과 결과 사이의 시간 | 제품 출시 → 시장 인식 (3개월 지연) |

### 실전 사례: 고객 보유 시스템

```
╔════════════════════════════════════════════════════╗
║             고객 보유 시스템 맵                     ║
╠════════════════════════════════════════════════════╣

Stock: 활성고객(1000명) ← Flow in: 신규가입(100/월)
                        ← Flow out: 이탈(50/월)

Feedback Loop 1 (강화 루프):
  활성고객 ↑ → 수익 ↑ → 제품투자 ↑ → 가치 ↑ → 신규가입 ↑
  
Feedback Loop 2 (균형 루프):
  활성고객 ↑ → 기술 부채 축적 ↑ → 버그 증가 ↑ → 이탈 ↑
  
Time Delay:
  제품 투자 → 출시(2개월) → 인식(1개월) → 행동(1개월) = 4개월 총 지연

Result:
  - 월간 신규가입 → 3개월 뒤 이탈로 반영
  - 제품 개선 → 분기 뒤 고객 만족도로 반영
```

### 지표 설계 함의

| 지표 설계 오류 | 시스템 다이내믹스 관점 |
|---------------|------------------|
| 신규가입(Flow in)만 추적 | Stock(활성고객) 감소 무시. 이탈 루프 있는지 확인 필수 |
| 단기 지표(1주)만 보기 | Delay 때문에 인과관계 보이지 않음. 최소 1분기 보기 |
| L2 드라이버만 추적 | 강화 루프와 균형 루프의 상호작용 놓침. L3·L4 필수 |

## 3. Goldratt의 제약이론 (Theory of Constraints)

### 핵심 아이디어

**시스템의 성과는 가장 약한 환(약점, constraint)에 의해 결정된다.**

비유: 체인의 강도는 가장 약한 링크로 결정됨.

### Throughput Accounting

기존 원가회계의 문제:

| 기존 원가회계 | 제약이론 관점 |
|------------|------------|
| "이 제품은 원가 $100, 판가 $150 → 이익 $50" | 잘못된 판단. 실제 이익은? |
| 각 부서의 비용 최적화 | 전체 시스템 처리량(throughput) 최적화 X |
| "효율성" 중심 (모든 자원을 바쁘게) | "효과성" 중심 (약점만 최대 활용) |

제약이론의 개념:

```
Throughput (T)
= 판매가 - 변동비(팔때마다 드는 비용)

Inventory (I)
= 현재 시스템에 묶여 있는 돈

Operating Expense (OE)
= 월간 고정비(급여, 임대료 등)

시스템 성과 = T / (I + OE)
```

### 5가지 포커싱 스텝 (5 Focusing Steps)

제약을 제거하는 절차:

| 스텝 | 행동 | 예시 |
|------|------|------|
| **1. Identify** | 약점(constraint) 찾기 | "병목은 마케팅 팀이다. 리드 생성이 최대 100/월" |
| **2. Exploit** | 약점을 최대로 활용 | "마케팅 효율을 X% 개선하면 리드 150/월 가능" |
| **3. Subordinate** | 다른 자원을 약점에 맞춤 | "영업팀도 100/월 리드만 처리 가능하도록 조정" |
| **4. Elevate** | 약점 제거 (투자) | "마케팅 자동화 도구 도입 → 처리 능력 300/월로" |
| **5. Repeat** | 새 약점 찾기 | "이제 약점은 영업 전환율. 반복" |

### 지표 설계 함의

| 오류 | 바른 대응 |
|-----|---------|
| 모든 부서의 효율성을 고르게 추적 | **약점 부서의 지표에 가중치.** 약점이 아닌 곳은 "충분하게만" |
| "각 팀이 최대 성과" 기대 | **약점에 자원 집중.** 다른 곳의 유휴 능력은 정상 |
| 전체 처리량(throughput) 지표 없음 | **T = 판매가 - 변동비.** 이것만 중요. 효율성 지표는 보조 |

## 4. 피드백 루프 매핑 프로토콜

### 절차

지표 설계 시점에, 각 L2 드라이버 간 **피드백 루프를 명시화하세요.**

#### 단계 1: 모든 L2 지표 쌍을 나열

예:
- 신규고객수
- 고객리텐션율
- 제품품질(버그수)
- 개발속도(릴리스/분기)

#### 단계 2: 각 쌍의 피드백 방향 결정

```
신규고객수 ↑
  → 리텐션율에 영향: ? (양 / 음 / 무관)

고객리텐션율 ↑
  → 제품품질에 영향: ? (양 / 음 / 무관)

개발속도 ↑
  → 신규고객수에 영향: ? (양 / 음 / 무관)
```

#### 단계 3: 루프 유형 분류

| 루프 | 이름 | 신호 | 효과 |
|------|------|------|------|
| **양(+) 피드백** | 강화 루프(Reinforcing) | A ↑ → B ↑ → A ↑ | 기하급수 성장 또는 기하급수 쇠퇴 |
| **음(-) 피드백** | 균형 루프(Balancing) | A ↑ → B ↓ → A ↓ | 목표 지향, 안정화 |

#### 단계 4: 지연(delay) 파악

```
신규고객 ↑ → 리텐션율 ↓?
  (지연: 3개월. 신규고객은 온보딩 미숙)

개발속도 ↑ → 버그수 ↑?
  (지연: 2주. 충분한 테스트 부족)

버그수 ↓ → 리텐션율 ↑?
  (지연: 1개월. 고객이 개선을 인지할 때까지)
```

### 시스템 안정성 판정

```
강화 루프만 있으면? → 통제 불가능 (폭주)
균형 루프만 있으면? → 너무 안정적 (성장 무뎀)
혼합? → 정상

예시:
  신규고객 → 수익 → 투자 (강화)
  신규고객 → 기술부채 → 이탈 (균형)
  → 균형이 강화를 이기면 성장 멈춤
```

### 지표 설계의 체크

- **약한 강화 루프 감지**: L2 지표에서 성장 신호 약함 → 강화 루프 약화 신호. L3로 "투자 부족" 감시
- **강한 균형 루프 감지**: L3에서 "이탈율" 급증 → 균형 루프가 강화를 압도. 급히 조정 필요

## 5. Meadows 12단계 적용 체크리스트

지표 설계 시, 각 지표가 몇 번 레버리지에 해당하는지 분류하세요:

| L2 드라이버 지표 | 레버리지 단계 | 단기 효과 | 장기 의존성 | 체크 |
|-----------------|-------------|---------|-----------|------|
| 신규고객수(마케팅 인센티브) | 1~2 (상수·버퍼) | 3개월 | 제품품질, 운영비용 의존 | △ 약함 |
| 제품출시주기(개발 속도) | 3~4 (정보흐름·규칙) | 6개월 | 아키텍처 설계 의존 | △ 보통 |
| 고객만족도(NPS) | 6 (피드백루프) | 장기 | 제품전략, 조직문화 의존 | ◎ 강함 |

**결론**: 높은 레버리지 포인트 지표가 없으면, 지표 개선이 시스템을 바꾸지 못한다.

## 6. 실전 예시: SaaS 성장 시스템

```
┌─────────────────────────────────────────────────────────────┐
│              SaaS 성장 시스템 맵                             │
└─────────────────────────────────────────────────────────────┘

Stock: MRR(월간반복수익)

Inflow 1: 신규고객 × 평균 가입금액
  └─ 드라이버: 마케팅 (L2-1)

Inflow 2: 기존고객 업셀 × 추가금액
  └─ 드라이버: 제품 가치 (L2-2)

Outflow: 이탈고객 × 평균 가입금액
  └─ 드라이버: 고객 만족도 (L2-3)

Feedback Loop 1 (강화):
  MRR ↑ → 투자 여력 ↑ → 개발속도 ↑ → 기능 ↑ → 만족도 ↑ → MRR ↑

Feedback Loop 2 (균형 - 위험):
  개발속도 ↑ → 버그 ↑ → 만족도 ↓ → 이탈 ↑ → MRR ↓

지표 체계:
  L1: MRR
  L2-1: 신규고객수/월
  L2-2: 기존고객 업셀금액/월
  L2-3: NPS 또는 고객 만족도
  L3: 버그수/주 (강화루프 방어)
  L3: 개발속도 (기술부채 모니터)
  L4: 이탈율 (역효과 감시)

의존성:
  - 신규고객수 ↑ → 기술부채 ↑ (3개월 지연) → 만족도 ↓ (2개월 지연)
  - 투자 2개월 지연 → MRR 4개월 지연 반영
    → 분기별 리뷰 필수, 주간 리뷰만으로는 인과관계 보이지 않음
```

## 요약: 시스템 건전성 체크

매분기 리뷰 시:

| 체크 | 질문 |
|------|------|
| **피드백 루프** | 강화/균형 루프가 어떤 상태? 균형이 강화를 압도? |
| **약점** | 현재 시스템의 약점은? (제약이론 5스텝) |
| **레버리지** | 추적하는 지표가 몇 번 레버리지? 높은 레버리지 부족? |
| **지연** | 지표 변화 → 실제 결과 반영까지 얼마나? |

**결론**: 지표가 시스템 다이내믹스를 반영하지 못하면, 아무리 정교한 대시보드도 오판을 초래한다.
