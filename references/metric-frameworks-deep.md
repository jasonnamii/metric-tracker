# 지표 설계 프레임워크 — BSC, OKR, AARRR, North Star, 선행/후행 계층

최고 수준의 지표 설계 체계들과 각각의 위험성, 그리고 올바른 적용 방법.

---

## 1. 균형성과표(Balanced Scorecard, BSC)

### 역사와 목적 (Kaplan & Norton, 1992)

BSC는 "재무 지표만으로는 부족하다"는 깨달음에서 시작. 4개 관점을 통해 전략을 구현하는 틀.

### 4개 관점과 인과 연쇄

```
학습과 성장 (Learning & Growth)
      ↓
    ↙   ↘
내부 프로세스   고객
      ↖   ↙
     재무
```

각 관점의 예시:

| 관점 | 정의 | L2 지표 예시 |
|------|------|-----------|
| **학습과 성장** | 조직 능력, 직원 역량, 기술 | 직원 만족도, 교육시간, 프로세스 효율성 |
| **내부 프로세스** | 핵심 운영 프로세스 | 개발속도, 제품 품질(버그), 출시 시간 |
| **고객** | 고객 만족과 보유 | NPS, 고객 만족도, 리텐션 |
| **재무** | 사업 성과 | 수익, 이익, ROI |

### 인과 연쇄의 핵심

> 직원 역량 ↑ → 프로세스 효율성 ↑ → 고객 만족 ↑ → 매출 ↑

이것이 "전략 지도(Strategy Map)"다. 각 관점의 지표가 인과관계를 가져야 의미 있음.

### 실전 위험: BSC를 잘못 적용하는 경우

| 오류 | 신호 | 결과 |
|------|------|------|
| **대시보드로 착각** | 4개 관점마다 3~5개 지표 = 12~20개 지표로 비대화 | 보기만 좋고 실행 불가. SKILL.md 절대규칙 #5 위반 |
| **인과관계 미정의** | "학습 지표 좋은데 고객 지표는 안 좋아도 상관없다" | 지표 간 인과관계가 없으면 BSC가 아니라 대시보드일 뿐 |
| **전략이 없는 BSC** | 4개 관점을 기계적으로 채움 | 비즈니스 전략과 무관한 지표만 쌓임 |

### 올바른 적용

1. **먼저 전략을 정의**: "우리는 X 전략으로 Y를 달성하려 한다"
2. **4개 관점에서 인과 경로를 그림**: 학습→프로세스→고객→재무 순서로 연쇄 명시
3. **각 관점에 3~5개만**: 4 × 3 = 12개가 최대
4. **주기적 재검증**: 인과관계가 여전히 성립하는가?

## 2. OKR (Objectives & Key Results)

### 원래 목적 (Andy Grove의 MBO 개선, 1990년대)

목표 관리(MBO)의 문제점:
- 목표가 너무 많음
- 목표와 업무 평가가 연결되어 게임화됨
- 목표 달성 여부만 보고, 도중 학습 무시

OKR의 철학:
- **Objective**: 정성적, 영감을 주는 방향 (예: "우리는 X 세상을 만든다")
- **Key Results**: 정량적, 객관적, 검증 가능 (예: "MAU 1M → 5M")

### 구조와 계층

```
회사 수준 OKR (2~3개)
  ↓
팀 수준 OKR (3~5개, 회사 OKR과 정렬)
  ↓
개인 수준? (여기서 오류 발생)
```

### OKR 5가지 흔한 실수 (Andy Grove 원저, Doerr의 재정리)

| 실수 | 신호 | 대응 |
|------|------|------|
| **1. 성과평가와 연결** | OKR 달성을 월급·보너스와 직결 | OKR은 전략 추적 도구. 평가와 분리 필수. (Campbell 법칙) |
| **2. 개인 OKR = 업무 리스트** | "내 OKR은 10개" 같은 발언 | 개인 OKR은 권장하지 않음. 팀 OKR만 사용 |
| **3. OKR을 목표로 취급** | "100% 달성을 목표로" | OKR의 목표 신뢰도는 60~70%. 100% 달성은 목표가 너무 쉬운 것 |
| **4. 모호한 목표** | "더 좋은 제품 만들기" 같은 Objective | 명사 + 동사 + 구체적 맥락. 예: "신규 시장에서 신뢰도 리더 위치 확보" |
| **5. Business-as-usual OKR** | 매해 같은 OKR | 혁신적 도전 OKR 70% + 운영 OKR 30%. 도전이 없으면 의미 없음 |

### OKR 구현 시간표 (현실)

| 분기 | 상태 | 신호 |
|------|------|------|
| **Q1** | 초기 운영 | 지표가 있으지만 팀이 게임화하지 않음 |
| **Q2** | 분화 시작 | "목표 달성이 평가와 연결되나?" 질문 시작 |
| **Q3** | 위기 | 일부 팀이 OKR을 낮춰 달성 가능하게 조정 |
| **Q4** | 안정 | 신뢰도 규칙("60% 달성이 good") 정착 |
| **Q5~** | 성숙 | 4분기 후부터 진정한 OKR 문화 형성 |

**결론**: 구현에 4~5분기 필요. 서두르지 말 것.

## 3. AARRR (Pirate Metrics, 2007)

### 5단계 고객 생명주기

```
Acquisition → Activation → Retention → Revenue → Referral
   (획득)     (활성화)     (유지)      (수익화)    (추천)
```

### 각 단계의 핵심 지표

| 단계 | 정의 | L2 지표 | 벤치마크 |
|------|------|--------|--------|
| **Acquisition** | 신규 고객 획득 | CAC(고객획득비), 가입수/월 | 낮을수록 좋음 |
| **Activation** | 제품의 가치를 경험하는 첫 순간 | 온보딩 완료율, 첫 X일 내 기본 기능 사용 | 40% 이상 |
| **Retention** | 장기간 고객 유지 | Day7, Day30, Month3 리텐션 | 업종별 상이하나 50% 이상 |
| **Revenue** | 수익화 (무료→유료 또는 추가 구매) | ARPU, LTV, 전환율 | LTV:CAC ≥ 3:1 |
| **Referral** | 기존 고객의 추천 | 바이럴 루프(k-factor), NPS | k > 0.1 (선택사항) |

### 최고 레버리지 단계

```
Acquisition: 싼 비용으로 많은 고객 → 중요도 보통
  └─ 문제: 나쁜 고객도 많음

Activation: 신규 고객이 가치를 경험 → 중요도 매우 높음
  └─ 문제: 온보딩 실패 = 나머지 4단계 무너짐

Retention: 고객 유지 → 중요도 최고
  └─ 이유: LTV의 90%는 리텐션에서 비롯
           CAC 회수 가능 여부 결정

Revenue: 수익화 → 중요도 높음 (단, 리텐션 전제)

Referral: 추천 → 중요도 낮음 (Retention 확보 후 추가)
```

### 실전 오류: AARRR을 역순으로 최적화

**흔한 오류**: "매출(Revenue)을 올려야 한다" → Revenue 지표만 추적

**문제**: 
- Activation 낮음 (온보딩 미흡) → 신규 고객은 들어와도 이탈
- Retention 낮음 → LTV 계산 불가능
- CAC > LTV (적자 비즈니스)

**올바른 순서**:
1. **Activation 확인**: 신규 고객의 40% 이상이 온보딩 완료?
2. **Retention 확보**: Month3 리텐션 50% 이상?
3. **Revenue 최적화**: 확보한 고객으로부터 수익화

**AARRR 체크리스트**:

```
Activation 낮음 + Revenue 높음
  → 온보딩 개선 우선순위 #1
  
Retention 낮음 + Acquisition 높음
  → 고객은 늘어도 이탈이 더 빠름
  → 신규 고객 확보 멈추고 리텐션 개선
  
LTV:CAC < 3:1
  → 수익 비즈니스 아님
  → 매출 확대 금지, 효율성 개선 필수
```

## 4. North Star 지표

### 정의

**고객에게 제공하는 핵심 가치를 단일 지표로 표현한 것.**

예시:

| 회사 | North Star | 의미 |
|------|-----------|------|
| **Airbnb** | Nights Booked | 고객이 예약한 숙박 수 = 시장 크기 × 고객 만족 |
| **Slack** | Messages Sent | 팀의 협력 강도 = 제품 가치 |
| **Netflix** | Hours Watched | 고객이 본 시간 = 만족도 신호 |
| **LinkedIn** | Engaged Users | 활발하게 사용하는 사용자 |

### 특성

| 특성 | 의미 | 확인 |
|------|------|------|
| **Core Value** | 회사의 존재 이유를 반영 | "우리가 고객에게 주는 가장 큰 가치는 무엇인가?" |
| **Leading Indicator** | 미래 성장 신호 | "이 지표가 올라가면 매출도 올라가는가?" |
| **Actionable** | 팀이 즉시 영향을 줄 수 있는 것 | "우리가 이 지표를 올릴 수 있는가?" |
| **Measurable** | 정기적으로 추적 가능 | "일일/주간으로 추적 가능한가?" |

### North Star + Input Metrics 구조

```
North Star Metric (1개)
  ↑
  └─ Input Metrics (3~5개)
      ├─ Acquisition (신규 고객)
      ├─ Engagement (사용 강도)
      ├─ Retention (고객 유지)
      └─ Monetization (수익화)
```

각 Input Metric은 팀이 직접 영향 줄 수 있는 것들.

### 함정: North Star 게임화

**사례**: Slack이 "Messages Sent"를 추적하면, 팀이 의미 없는 메시지를 보냄.

**방어**: 
- 정기 감사: "메시지의 질은?"
- 카운터 지표: "mute된 채널 수" 추적
- 맥락: "유료 계정의 메시지만 센다" 같은 명확한 정의

## 5. Hubbard의 How to Measure Anything (AIE 프레임)

### 핵심 철학

> 측정 = 불확실성을 감소시키는 것

"측정할 수 없다"는 것은 오류. 모든 것은 측정 가능. 단지 방법이 다를 뿐.

### AIE 프레임 (Applied Information Economics)

```
Uncertainty (불확실성)
    ↓
Information Gap (정보 격차를 파악)
    ↓
Measurement (관찰 가능한 지표로 전환)
    ↓
Economics (측정 비용 vs 정보 가치 평가)
```

### 측정할 수 없어 보이는 것을 지표로 만드는 방법

**예: "신뢰도(Trust)" 측정**

```
1. 신뢰도를 정의 (추상 → 구체)
   신뢰도 = "고객이 제품 오류 후 계속 사용하는 정도"

2. 지표로 분해 (관찰 가능하게)
   ├─ 에러 후 재가입 확률
   ├─ 에러 후 환불 요청 비율
   ├─ NPS 점수 추세
   └─ 고객 인터뷰 정성 피드백

3. 측정 (정량 + 정성)
   정량: (재가입율 × 0.5) + (환불거절율 × 0.5)
   정성: 월간 10명 고객 인터뷰 + 신뢰도 5점 평가

4. 경제 평가
   "신뢰도 1점 개선 = 이탈율 X% 감소 = 매출 $Y 증가"
```

### McNamara 오류 피하기

Hubbard 방식의 핵심: **정량만이 아니라 정성도 함께.**

실수: "신뢰도를 측정할 수 없으니까 무시하자" → McNamara 함정

올바른 방식: "신뢰도를 정량(재가입율) + 정성(인터뷰)로 추정하자"

## 6. 선행/후행 계층 (Leading/Lagging Hierarchy)

### 5단계 인과 계층

```
Input → Activity → Output → Outcome → Impact
```

각 단계마다 "선행(Leading)" 또는 "후행(Lagging)" 여부가 달라짐:

| 단계 | 정의 | 예시 (영업팀) | 선행? | 후행? |
|------|------|-------------|-------|-------|
| **Input** | 시스템에 투입되는 자원 | 영업팀 규모, 마케팅 예산 | Leading (다음 단계 예측) | Lagging (자원 결정 후) |
| **Activity** | 실행된 행동 | 고객 회의 횟수, 제안서 작성 수 | Leading | Lagging |
| **Output** | 활동의 직접 결과 | 제안서 발송 건수 | Leading | Lagging |
| **Outcome** | 비즈니스에 미치는 결과 | 계약 체결, 매출 | Lagging | - |
| **Impact** | 장기 전략 실현 | 시장점유율, 브랜드 가치 | Ultra-Lagging | - |

### 흔한 오류: Output을 Outcome으로 착각

| 오류 | 신호 | 결과 |
|------|------|------|
| **Output만 추적** | "제안서 50개 발송했다!" | 계약은 5개 (10% 전환율) |
| **Outcome 수치화 안 함** | "계약 5개, 매출 $500k" (숨겨진 정보) | 진정한 성과 불명확 |
| **계층 건너뛰기** | "영업팀 규모 ↑ 하면 매출 ↑" 직접 인과 가정 | 중간의 Activity/Output 악화 무시 |

### 올바른 지표 구조

```
L1 (Impact): 매출 (최상위)
  ←
L2-1 (Outcome): 계약 건수/분기 (선행)
  ←
L2-2 (Output): 제안서 발송 수/월 (선행)
  ←
L2-3 (Activity): 고객 미팅/주 (선행)
  ←
L3 (Health): 제안서 승률 (후행)
  ←
L4 (Counter): 고객 불만족율 (후행)
```

각 L2 드라이버는 이전 단계의 "선행 지표"이면서 다음 단계의 "후행 지표"다.

## 요약: 프레임워크 선택 가이드

| 상황 | 추천 프레임 | 이유 |
|------|-----------|------|
| **전략 실행이 목표** | Balanced Scorecard | 4개 관점 인과 연쇄 명시 |
| **사업 목표 정렬이 목표** | OKR | 회사→팀 계층 정렬. 60% 신뢰도 원칙 |
| **성장 단계 진단** | AARRR | 고객 생명주기의 어느 단계가 약한지 파악 |
| **가장 중요한 것 1개** | North Star | 팀 전체가 같은 목표 추진 |
| **측정 불가능해 보이는 것** | Hubbard AIE | 정성 + 정량으로 포괄적 측정 |
| **인과관계 체크** | Leading/Lagging | 각 지표의 "언제 영향을 주는가" 명확히 |

**원칙**: 프레임을 섞지 말 것. 회사는 1개만 선택. (BSC 또는 OKR, 둘 다 아님)
